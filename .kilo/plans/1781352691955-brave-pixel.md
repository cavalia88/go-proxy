# Fix MiniMax M3 Tool-Calling / Schema Errors

## Problem

When `go-proxy` routes requests to `minimax-m3` (Anthropic endpoint via OpenCode Go), Kilocode frequently reports:

- `SchemaError(Missing key at ["command"])` — tool-call arguments are missing required parameters.
- `Model tried to call unavailable tool 'invalid'` — the model emits a tool-call name that is not in the configured tools list.

These errors must be fixed **only for `minimax-m3`** without changing response processing for any other model.

## Root-Cause Analysis

MiniMax M3 has known quirks compared with the other Anthropic-routed models (`minimax-m2.5`, `minimax-m2.7`, `qwen3.7-max`):

1. **Interleaved reasoning bleeds into tool arguments.** M3 natively performs reasoning inline. If the upstream does not split reasoning from tool JSON, raw thinking text can be merged into `tool_use.input`, producing malformed or incomplete arguments.
2. **Loose / generic JSON schemas confuse M3.** M3 is reported to struggle with `type: object` parameters that do not explicitly enumerate their properties. It may emit `{}`, `null`, or omit required fields such as `command`.
3. **Hallucinated / malformed tool names.** When schema guidance is weak, M3 may invent a tool name such as `invalid` or fail to map its native output to the exact names supplied in `tools`.
4. **Response sanitization is model-agnostic today.** `TransformAnthropicToOpenAI` and the stream transformer forward every `tool_use` block as-is, so any upstream malformed tool call reaches Kilocode and triggers schema validation errors.

`go-proxy` currently treats all Anthropic-endpoint models identically. The fix therefore requires **model-gated transformations** that apply only when `model_id == "minimax-m3"`.

## Design Principles

- **Scope isolation:** All M3-specific behavior is gated by an explicit `IsMiniMaxM3(modelID)` helper. No other model's request or response path changes.
- **Minimal intrusion:** Existing `TransformOpenAIToAnthropic`, `TransformAnthropicToOpenAI`, and stream transformer logic remain the default. M3-specific code is additive and branches only when needed.
- **Graceful degradation:** If a mitigation cannot be applied safely, fall back to the current behavior rather than breaking the request.
- **User-visible toggle:** Each major mitigation is configurable per-model so users can disable one without disabling the others.

## Proposed Mitigations

### 1. Request Hardening (OpenAI → Anthropic)

When the model is `minimax-m3`:

1. **Inject a strict system prompt.** Append a short, imperative instruction to the system text:
   ```text
   CRITICAL: When calling a tool, you must strictly populate the argument parameters exactly matching the provided JSON schema. Do not include any conversational filler, markdown formatting inside the arguments, or unprompted keys. If a parameter is required, you must provide a valid value for it. Only use tool names that are explicitly listed in the available tools.
   ```
   If the request has no system message, create one.
2. **Tighten tool schemas.** Walk each tool's `parameters` JSON schema. For every `type: object` that lacks explicit `properties`, replace it with a strict array-of-key-value representation or add `"additionalProperties": false` and empty `properties` where semantically safe. Prefer minimal schema rewriting that preserves the original intent.
3. **Add `reasoning_split: true` if the upstream supports it.** Extend `MessageRequest` with an optional `ExtraBody json.RawMessage` field. For M3 only, marshal `{"reasoning_split": true}` into the request JSON at the top level. This is gated by a config flag so it can be disabled if OpenCode Go's Anthropic endpoint rejects it.

### 2. Response Sanitization (Anthropic → OpenAI)

When the model is `minimax-m3`:

1. **Validate tool names.** After `TransformAnthropicToOpenAI`, drop any `tool_call` whose `function.name` is not present in the original request's `tools` list. Convert dropped tool calls to plain text content that tells the model the tool was invalid, preserving a recoverable response.
2. **Validate required arguments.** For each surviving tool call, compare its parsed `arguments` object against the tool's JSON schema. If a required key is missing:
   - Fill it with a sensible default when possible (`""` for strings, `0` for numbers, `[]` for arrays, `{}` for objects), **or**
   - Drop the tool call and convert it to text if the missing value cannot be defaulted safely.
3. **Repair malformed JSON arguments.** If `function.arguments` is not valid JSON, attempt to extract JSON from markdown code fences or trailing text. If repair fails, default to `{}` or drop the call.

For streaming, the same validation runs on the accumulated response state after the stream ends (and optionally on each `content_block_stop` for `tool_use` blocks). The sanitized deltas are emitted downstream.

### 3. Optional Config Knobs

Add the following optional fields to `ModelConfig`:

```json
{
  "minimax-m3": {
    "model_id": "minimax-m3",
    "endpoint": "anthropic",
    "m3_strict_prompt": true,
    "m3_tighten_schemas": true,
    "m3_reasoning_split": true,
    "m3_validate_tool_calls": true
  }
}
```

All flags default to `true` for `minimax-m3` if omitted, and to `false` for every other model. This keeps the fix self-contained and discoverable.

## Implementation Plan

### Phase 1 — Config & Helpers

1. **Extend `internal/config/config.go`.**
   - Add `M3StrictPrompt`, `M3TightenSchemas`, `M3ReasoningSplit`, `M3ValidateToolCalls` bool fields to `ModelConfig`.
2. **Add `internal/transformer/minimax_m3.go`.**
   - `IsMiniMaxM3(modelID string) bool`
   - `ApplyM3RequestHardening(req *types.MessageRequest, originalTools []types.ToolDef, cfg config.ModelConfig) error`
   - `SanitizeM3Response(resp *types.ChatCompletionResponse, originalTools []types.ToolDef, cfg config.ModelConfig) error`
   - `SanitizeM3StreamResponse(resp *types.MessageResponse, originalTools []types.ToolDef, cfg config.ModelConfig) error`
   - Helper: `tightenSchema(schema json.RawMessage) json.RawMessage`
   - Helper: `validateToolCall(toolCall *types.ToolCall, toolDefs []types.ToolDef) (bool, string)` — returns (ok, replacementText)

### Phase 2 — Request Path

1. **Update `internal/handlers/completions.go` `handleAnthropicRequest`.**
   - Keep the original `openaiReq` and `modelConfig` accessible.
   - After `TransformOpenAIToAnthropic`, call `transformer.ApplyM3RequestHardening(anthropicReq, openaiReq.Tools, modelConfig)` only when `IsMiniMaxM3(modelConfig.ModelID)`.
2. **Update `internal/pkg/types/anthropic.go`.**
   - Add `ExtraBody json.RawMessage` to `MessageRequest` with `json:"-"` so it is merged into the final JSON only when set, avoiding breakage for non-M3 models.

### Phase 3 — Response Path

1. **Update `internal/transformer/reverse_response.go` `TransformAnthropicToOpenAI`.**
   - Add `originalTools []types.ToolDef` and `cfg config.ModelConfig` parameters, or introduce a wrapper `TransformAnthropicToOpenAIForModel` that calls the original function then applies M3 sanitization only for M3.
   - Recommended: keep the original signature untouched and add a new exported function `TransformAnthropicToOpenAIWithConfig(resp, model, tools, cfg)` that the handler calls. The original function becomes a thin wrapper for backward compatibility.
2. **Update `internal/handlers/completions.go` `handleAnthropicNonStreamingResponse`.**
   - Pass `openaiReq.Tools` and `modelConfig` to the new transform function.
3. **Update streaming path.**
   - Extend `AnthropicStreamTransformer` with `originalTools` and `modelConfig`.
   - After `TransformStream` returns, call `SanitizeM3StreamResponse` on the accumulated `t.response` if the model is M3. Emit any corrections as a final SSE chunk before `[DONE]`.

### Phase 4 — Tests

1. **Add `internal/transformer/minimax_m3_test.go`.**
   - Test that strict prompt is injected only for M3.
   - Test schema tightening preserves required fields and rejects loose objects.
   - Test that invalid tool names are dropped/converted.
   - Test that missing required arguments are filled or dropped.
   - Test that non-M3 requests are unchanged.
2. **Update `internal/handlers/completions.go` tests if they exist.** (No handler tests exist today; add focused tests only if time permits.)
3. **Run existing test suite:** `go test ./...`

### Phase 5 — Documentation

1. **Update `MODELS.md`.**
   - Document the four M3-specific config flags and their defaults.
   - Add a short troubleshooting section for M3 tool-call errors.
2. **Update `configs/config.example.json`.**
   - Include the `minimax-m3` entry with the new flags commented/explained.

## Risks & Mitigations

| Risk | Mitigation |
|------|------------|
| `reasoning_split` is rejected by OpenCode Go. | Flag defaults to `true` but can be disabled in config; if upstream returns 400, log the error and auto-disable for that request. |
| Schema tightening changes tool semantics. | Only rewrite loose `type: object` nodes; preserve explicit schemas unchanged. Add unit tests for every transformation. |
| Dropping invalid tool calls loses information. | Convert dropped calls to assistant text so the user/model can recover. |
| Response sanitization adds latency. | Validation is O(number of tools × number of tool calls), which is negligible for typical agent turns. |
